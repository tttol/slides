# 1. 現状分析（スタート）

---

# まずコードベースを読む

`git clone` してテストコードの実装状況を確認するところから始める。

<br>

テストがゼロの場合
- コードベースからわかることはあまりない
- 代わりに「どんなテストをしているか」を確認する
- Playwright / Selenium / マニュアルテスト など
  何かしらのE2Eテストをしていることが多い

---

# テストがある場合の確認ポイント

テストコードはあるが、正しく機能しているかを確認する。

<br>

自分が実際に経験したパターン：

| パターン | 内容 |
|---|---|
| 一部だけ | 一部の機能にだけテストがある |
| 実行エラー | テストを実行するとエラーになる |
| UTのみ | Unit Testはあるが Integration Testはない |
| CI未整備 | テストはあるがCIで自動実行されていない |

---

# 現状分析のゴール

この段階では判断・提案はしない。

<br>

- プロジェクトのテスト状況を正確に把握することが目的
- 「なぜそうなったのか」は次のステップで考える
- まずは観察者に徹する

---

<!-- _class: title -->

# 2. テストがない/少ない原因の解明

---

# まず切り分けるポイント

<br>

```
「意図的にテストを書かない選択をしている」
              vs
「書きたいけど何らかの事情で書けない」
```

<br>

このどちらかによって、次のアクションが大きく変わる。

---

# ケース①：意図的にテストを書かない

よくある理由：

- 「テストを書く時間がない」
- 「テストを書く必要性がわからない」

<br>

→ テストがあることの具体的なメリットを伝える必要がある

---

# テストが与えてくれる「気づき」

機能Aをリファクタリングしてテストを修正したら、
全然関係のない機能Bのテストが落ちた。

<br>

→ Aへのリファクタが間接的にBにも影響していた！

<br>

もしテストがなかったら？
Bへの影響に気づかないままリリースして、
後から不具合が発覚してまた対応する。
開発リードタイムは延びていく。

---

# テストがあることのメリット（整理）

<br>

| メリット | 説明 |
|---|---|
| デグレ検知 | 意図しない影響範囲の変更を早期発見 |
| リードタイム短縮 | 後から不具合対応するコストを削減 |
| 仕様の文書化 | テストがコードの振る舞いを説明する |
| リファクタの安心感 | テストがあればコードを大胆に改善できる |

---

# ケース②：書きたいけど書けない

よくある事情：

- テストを書くスキルがない・自信がない
- テスト用の環境が整っていない（DBのセットアップなど）
- 納期・工数の制約で後回しにせざるを得なかった
- レガシーコードでテストが書きにくい構造になっている

<br>

→ 技術的なサポートと環境整備が必要

---

<!-- _class: title -->

# 3. テスト強化をチームに提案する

---

# 「テストを書きましょう！」だけではダメ

提案の言い方・立ち回り方が最も重要で、
最も慎重に振る舞うべきポイント。

<br>

まず大前提として：

既存コードベースに敬意を払う

---

# 既存コードベースに敬意を払う

現在のコードがどんな状態でも、
そのコードを何ヶ月・何年も面倒みてきたメンバーを
批判する理由にはならない。

<br>

ビジネス上の事情・納期の事情など様々な理由で
ベストプラクティスから逸れた実装をせざるを得ない
シーンはよくある。

---

# 過去を知る

- コミットログ
- チケット・Issue
- Slackの履歴

などから情報を得て、コードの歴史を知る。

<br>

彼らが過去にどんな苦労をしてきたのかを
理解した上で提案することが、
チームの信頼を得る第一歩。

---

# テスト強化の訴求ポイント

敬意を払うだけでは動かない。
具体的なメリットも同時に伝える。

<br>

| 訴求ポイント | 補足 |
|---|---|
| マニュアルテストが自動テストに置き換わる | 作業が増えるのではなく置き換わる |
| 開発リードタイムが縮まる | デグレ対応コストの削減 |

<br>

特に「作業が増えるのではなく置き換わる」は
重要なフレーミング。

---

# 立ち回りのポイント①

最初から全機能のテストを書こうとしない

<br>

- ハッピーパス（正常系）だけに絞ってテストを書く
- 異常系・エッジケースは時間とともに少しずつ増やす
- 「完璧なテスト」より「動くテストが1本ある」状態を目指す

---

# 立ち回りのポイント②

テストの実装は自分もやる、むしろ自分がやる

<br>

- 「実装してください」と言うだけでは弱い
- 自ら実装して効果を確かめて見せる
- チームメンバーが「テストって便利だな」と
  実感できる体験を作ることが目的

---

<!-- _class: title -->

# 4. テストを実装する

---

# 既存テストがある場合

まず既存のテストが正しく動いているかを確認する。

<br>

チェックポイント：
1. テスト実行結果は OK / NG どちらか
2. OKだとしてプロダクトコードを正しく検証できているか
3. 意味のないアサーションになっていないか

<br>

あまりにひどいテストは
思い切って削除するのも一つの選択肢。

---

# テストがゼロの場合

まずハッピーパスを満たすテストを1本書く。

<br>

- 細かいエッジケースは後回しでよい
- 1本書くことで「テストが実行できる環境」が整う
- その1本が次のテストを書く足がかりになる

---

# 最初に書くテストは「古典学派」スタイルで

古典学派（Classic School）とは：

モックを極力使わず、
テスト対象メソッドの振る舞いを
可能な限り全て実際に実行するスタイル

<br>

対比：ロンドン学派（London School）
→ モックを積極的に使い、関数単位でUTを書くスタイル

---

# なぜ最初は古典学派なのか

ロンドン学派のテストでは：
- 関数単位のUTが出来上がるだけで恩恵を感じにくい
- モックを多用すると実装の詳細に依存しやすい
- 偽陽性（テストはパスするが実際は壊れている）を生みやすい

<br>

古典学派のテストは：
- 振る舞いベースのためリファクタリングに強い
- 偽陽性のリスクが低い
- マニュアルテストの自動化として明快な恩恵がある

---

# 古典学派テストの具体例

テスト対象のコントローラ：

```java
@RestController
@RequiredArgsConstructor
public class SomeController {

    private final HogeService hogeService;

    public ResponseEntity<Voidindex() {
        hogeService.methodA();
        hogeService.methodB();
        hogeService.methodC();
        return ResponseEntity.ok().build();
    }
}
```

---

# ロンドン学派アプローチ（避けたい）

```java
@Test
void index_ロンドン学派() {
    // HogeServiceをモック化
    HogeService mockService = mock(HogeService.class);
    SomeController controller = new SomeController(mockService);

    controller.index();

    // メソッドが呼ばれたかだけを検証
    verify(mockService).methodA();
    verify(mockService).methodB();
    verify(mockService).methodC();
}
```

→ 実装の詳細に依存、リファクタで壊れやすい

---

# 古典学派アプローチ（推奨）

```java
@SpringBootTest
@AutoConfigureMockMvc
class SomeControllerTest {

    @Autowired
    private MockMvc mockMvc;

    @Test
    void index_正常系() throws Exception {
        // GIVEN: テスト用DBにデータを準備

        // WHEN: 実際のエンドポイントを呼び出す
        mockMvc.perform(get("/"))
            // THEN: レスポンスと副作用（DB状態）を検証
            .andExpect(status().isOk());
        // DB上の状態も検証する
    }
}
```

---

# 古典学派の原則

| 項目 | 方針 |
|---|---|
| モックの使用 | 極力使わない |
| DB | テスト用の本物のDBを用意して実際にアクセス |
| 外部サービスのAPI | モックを使う（例外） |
| AWSなどのクラウドリソース | モックを使う（例外） |

<br>

コントローラ単位の振る舞いテストは
マニュアルテストの自動化として非常に明快。

---

# テストカバレッジについて

カバレッジはほどほどに気にするのが良い。

<br>

- 目安：85〜90% を数値目標として掲げる
- 100%を目指す必要はない

---

# なぜ100%を目指さないのか

理由①：90%→100%への道のりが険しくて旨みがない

- `IOException` の catch節
- 業務上通ることのない else ルート
- コアロジックとは関係のない箇所

→ これらに対してテストを書いても、コストに見合わない

---

# なぜ100%を目指さないのか

理由②：カバレッジは「認知しているケース」にしか効かない

<br>

methodA / B / C のテストを書いたが、
実は業務的に methodD が必要だった！

<br>

→ テストコードは A/B/C をカバーしているので 100% になるが、
　 実際には全ケースを網羅できていない。

<br>

カバレッジ = 自分が知っているケースへの保険

---

<!-- _class: title -->

# 5. CIによる自動化

---

# CIによるテストの自動実行は必須

テストは自動実行されてこそ輝く。

<br>

自動で実行されないテストは
いつか必ず腐る。

<br>

開発者の手動実行に依存する運用は、
そのうち誰も実行しなくなる。

---

# 自動実行のタイミング

一般的には PRが作成された時点で自動実行されるべき。

<br>

| ツール | 選択肢 |
|---|---|
| GitHub | GitHub Actions |
| GitLab | GitLab CI |
| AWS | CodeBuild |

どのツールでも構わない。
大事なのは自動で動く仕組みを作ること。

---

# 自動実行の結果は必ず通知する

テストが失敗したことに気づかずに（あるいは見て見ぬふりをして）
開発を進めることを防ぐ。

<br>

GitHub Actionsであれば：
- CIが通っていないPRをマージさせない設定も可能
- Slack通知でテスト失敗を即時共有

<br>

→ 失敗を無視できない環境を作ることが重要。

---

# CI整備で得られること

<br>

| Before（CI未整備） | After（CI整備） |
|---|---|
| テスト実行を忘れる | 自動で必ず実行される |
| 失敗に気づかない | 即時通知される |
| テストが腐る | テストの健全性が保たれる |
| 属人的な運用 | チーム全体で管理 |

---

<!-- _class: title -->

# 6. 小さな成功体験を作る

---

# 「テストがあってよかった！」体験を作る

大きなものでなくていい。
テストを正しく実装していれば、
そのうち必ずこの体験は訪れる。

<br>

具体例：
- 機能改修時にデグレに気づくことができた
- マニュアルテストの対応コストが減った

---

# 成功体験がチームを変える

成功体験をメンバーが実感することで：

- テスト実装へのモチベーションが高まる
- 「テストって意味あるんだ」という認識が広がる
- テストを書くことが文化になっていく

<br>

この体験が「根付かせる」ための
最大の推進力になる。

---

<!-- _class: title -->

# 7. テストを書く習慣がつくまで見守る

---

# 習慣が浸透するまで見守る

これは言い換えると：
テストをサボるメンバーを指摘するということ。

<br>

よく起きること：

「忙しいのでテストは後で書きます」
→ そのまま書かれない

---

# 忙しさはテストをサボる理由にならない

個人的な考え：

テストを書かない方が最終的な開発リードタイムは延びる。
したがって、忙しさはテストをサボる理由にはならない。

<br>

ただし、プロジェクトの状況によっては
一分一秒すら惜しいこともある。

---

# 例外的にサボることを許容する場合

メンバーと相談して、やむを得ない事情がある場合は
テストなし（あるいはテストをコメントアウト）で
マージすることも許容する。

<br>

ただし必ず条件をつける：

- サボったテストはチケットなどで必ず管理する
- 後から必ず実装する約束をとりつける

---

# テスト文化が崩れないように

サボることを無条件に許容してしまうと：

<br>

```
テストをサボる
    ↓
その文化がじわじわ広がる
    ↓
またテストのないプロジェクトに戻る
```

<br>

テストを書く習慣が浸透するまで
じっくり見守っていく必要がある。

---

<br>

<br>

<br>

これがなければ、どれだけ正しい提案をしても
チームは動かない。

# Appendix

---

# 古典学派 vs ロンドン学派

| | 古典学派 | ロンドン学派 |
|---|---|---|
| モックの使用 | 極力使わない | 積極的に使う |
| テスト粒度 | 振る舞い単位 | 関数単位 |
| リファクタ耐性 | 高い | 低い（壊れやすい） |
| 偽陽性リスク | 低い | 高い |
| 導入コスト | 環境準備が必要 | 低い |

どちらが優れているかではなく、状況に応じて使い分ける。

---

# テストコードのないプロジェクトへの参画

よくある心理：

「既存コードがひどすぎる…全部書き直したい」

この気持ちはとてもわかる。
しかし、まずは既存コードへの理解と敬意が先。

<br>

コードが今の状態になった背景には
必ずそれなりの事情がある。

---

# テストを書く文化の定着に必要な時間

経験上：

- 最初の成功体験まで：1〜3ヶ月
- チームに習慣が定着するまで：6ヶ月〜1年

<br>

焦らず、着実に。
小さな成功体験の積み重ねが文化を作る。
