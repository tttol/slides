# このセッションで話すこと

テストがない（機能していない）プロジェクトに参画したとき、あなたならどうしますか？

<br>

- テストを導入するための技術的なステップ
- チームの合意を得るための対話の進め方

を共有します。

<br>

※本セッションで言及するテストとはJUnitなどのテストフレームワークによって実装されるテストコードを指しています。

---

1. 現状分析
2. テストがない原因の解明
3. テスト強化を提案する
4. テストを実装する
5. CIによる自動化
6. 小さな成功体験を作る（ゴール）
7. テストを書く習慣がつくまで見守る

<!-- _class: title -->

# 1. 現状分析（スタート）

---

# まずコードベースを読む

`git clone` してテストコードの実装状況を確認するところから始める。

<br>

テストがゼロの場合
- コードベースからわかることはあまりない
- 代わりに「どんなテストをしているか」を確認する
- Playwright / Selenium / マニュアルテストなど何かしらのE2Eテスト

---

# テストがある場合の確認ポイント

テストコードはあるが、正しく機能しているかを確認する。

<br>

| パターン | 内容 |
|---|---|
| 一部だけ | 一部の機能にだけテストがある |
| 実行エラー | テストを実行するとエラーになる |
| UTのみ | Unit Testはあるが Integration Testはない |
| CI未整備 | テストはあるがCIで自動実行されていない |

<br>

この段階では判断・提案はしない。まずは観察者に徹する。

---

<!-- _class: title -->

# 2. テストがない/少ない原因の解明

---

# まず切り分けるポイント

<br>

```
「意図的にテストを書かない選択をしている」
              vs
「書きたいけど何らかの事情で書けない」
```

<br>

このどちらかによって、次のアクションが大きく変わる。

---

# ケース①：意図的にテストを書かない

よくある理由：「テストを書く時間がない」「テストを書く必要性がわからない」

→ テストがあることの具体的なメリットを伝える必要がある

<br>

機能Aをリファクタリングしたら、関係のない機能Bのテストが落ちた。
→ Aへのリファクタが間接的にBにも影響していた！

テストがなかったら？→ Bへの影響に気づかないままリリースして不具合が発覚。
開発リードタイムは延びていく。

---

# テストがあることのメリット（整理）

<br>

| メリット | 説明 |
|---|---|
| デグレ検知 | 意図しない影響範囲の変更を早期発見 |
| リードタイム短縮 | 後から不具合対応するコストを削減 |
| 仕様の文書化 | テストがコードの振る舞いを説明する |
| リファクタの安心感 | テストがあればコードを大胆に改善できる |

---

# ケース②：書きたいけど書けない

よくある事情：

- テストを書くスキルがない・自信がない
- テスト用の環境が整っていない（DBのセットアップなど）
- 納期・工数の制約で後回しにせざるを得なかった
- レガシーコードでテストが書きにくい構造になっている

<br>

→ 技術的なサポートと環境整備が必要

---

<!-- _class: title -->

# 3. テスト強化をチームに提案する

---

# 提案の前に：既存コードベースへの敬意

「テストを書きましょう！」だけでは動かない。提案の立ち回り方が最も重要。

<br>

まず大前提として：**既存コードベースに敬意を払う**

現在のコードがどんな状態でも、何ヶ月・何年も面倒みてきたメンバーを批判する理由にはならない。コミットログ・チケット・Slackの履歴などから歴史を知り、理解した上で提案することがチームの信頼を得る第一歩。

---

# テスト強化の訴求ポイント

敬意を払うだけでは動かない。具体的なメリットも同時に伝える。

<br>

| 訴求ポイント | 補足 |
|---|---|
| マニュアルテストが自動テストに置き換わる | 作業が増えるのではなく置き換わる |
| 開発リードタイムが縮まる | デグレ対応コストの削減 |

<br>

特に「作業が増えるのではなく置き換わる」は重要なフレーミング。

---

# 立ち回りのポイント

**① 最初から全機能のテストを書こうとしない**
- ハッピーパス（正常系）だけに絞る
- 「完璧なテスト」より「動くテストが1本ある」状態を目指す

<br>

**② テストの実装は自分もやる、むしろ自分がやる**
- 「実装してください」と言うだけでは弱い
- 自ら実装して効果を確かめて見せる
- チームメンバーが「テストって便利だな」と実感できる体験を作る

---

<!-- _class: title -->

# 4. テストを実装する

---

# 既存テストがある場合

まず既存のテストが正しく動いているかを確認する。

<br>

チェックポイント：
1. テスト実行結果は OK / NG どちらか
2. OKだとしてプロダクトコードを正しく検証できているか
3. 意味のないアサーションになっていないか

<br>

あまりにひどいテストは思い切って削除するのも一つの選択肢。

---

# テストがゼロの場合 ＆ 最初のスタイル

まずハッピーパスを満たすテストを1本書く。
- 1本書くことで「テストが実行できる環境」が整う
- その1本が次のテストを書く足がかりになる

<br>

**最初は「古典学派」スタイルで書く**

モックを極力使わず、テスト対象の振る舞いを可能な限り実際に実行するスタイル。

- 振る舞いベースのためリファクタリングに強い・偽陽性のリスクが低い
- マニュアルテストの自動化として明快な恩恵がある

対比：ロンドン学派 → モックを積極的に使い関数単位でUTを書くスタイル（実装詳細に依存しやすく偽陽性を生みやすい）

---

# 古典学派テストの具体例

テスト対象のコントローラ：

```java
@RestController
@RequiredArgsConstructor
public class SomeController {

    private final HogeService hogeService;

    public ResponseEntity<Void> index() {
        hogeService.methodA();
        hogeService.methodB();
        hogeService.methodC();
        return ResponseEntity.ok().build();
    }
}
```

---

# ロンドン学派（避けたい） vs 古典学派（推奨）

```java
// ロンドン学派：実装詳細に依存、リファクタで壊れやすい
void index_ロンドン学派() {
    HogeService mockService = mock(HogeService.class);
    new SomeController(mockService).index();
    verify(mockService).methodA();
    verify(mockService).methodB();
}
```

```java
// 古典学派：振る舞いを検証、リファクタに強い
@SpringBootTest @AutoConfigureMockMvc
class SomeControllerTest {
    @Autowired MockMvc mockMvc;

    @Test
    void index_正常系() throws Exception {
        // GIVEN: テスト用DBにデータを準備
        // WHEN / THEN
        mockMvc.perform(get("/")).andExpect(status().isOk());
        // DB上の状態も検証する
    }
}
```

---

# 古典学派の原則 ＆ テストカバレッジ

| 項目 | 方針 |
|---|---|
| モックの使用 | 極力使わない |
| DB | テスト用の本物のDBを用意して実際にアクセス |
| 外部サービス・クラウドリソース | モックを使う（例外） |

<br>

**カバレッジは85〜90%を目安に。100%は目指さない。**

- `IOException` のcatch節・業務上通らないelseルートなどはコストに見合わない
- カバレッジは「認知しているケース」にしか効かない
  → methodD が必要なのにA/B/Cのテストだけ書いても100%になるが網羅できていない

# 5. CIによる自動化

---

# CIによるテストの自動実行は必須

テストは自動実行されてこそ輝く。自動で実行されないテストはいつか必ず腐る。

<br>

PRが作成された時点で自動実行されるべき。

| ツール | 選択肢 |
|---|---|
| GitHub | GitHub Actions |
| GitLab | GitLab CI |
| AWS | CodeBuild |

---

# 自動実行の通知 ＆ CI整備で得られること

失敗を無視できない環境を作ることが重要。

- CIが通っていないPRをマージさせない設定も可能
- Slack通知でテスト失敗を即時共有

<br>

| Before（CI未整備） | After（CI整備） |
|---|---|
| テスト実行を忘れる | 自動で必ず実行される |
| 失敗に気づかない | 即時通知される |
| テストが腐る | テストの健全性が保たれる |

---

<!-- _class: title -->

# 6. 小さな成功体験を作る

---

# 「テストがあってよかった！」体験を作る

大きなものでなくていい。テストを正しく実装していれば、そのうち必ずこの体験は訪れる。

具体例：
- 機能改修時にデグレに気づくことができた
- マニュアルテストの対応コストが減った

<br>

成功体験をメンバーが実感することで：
- テスト実装へのモチベーションが高まる
- テストを書くことが文化になっていく

この体験が「根付かせる」ための最大の推進力になる。

---

# 7. テストを書く習慣がつくまで見守る

習慣が浸透するまで見守る＝テストをサボるメンバーを指摘するということ。

「忙しいのでテストは後で書きます」→ そのまま書かれない

<br>

テストを書かない方が最終的な開発リードタイムは延びる。
→ 忙しさはテストをサボる理由にはならない。

やむを得ない場合は許容するが、必ず条件をつける：
- サボったテストはチケットなどで必ず管理する
- 後から必ず実装する約束をとりつける

無条件に許容すると→「またテストのないプロジェクトに戻る」

# まとめ

# テストコードが根付くまでのステップ（再掲）

| ステップ | ポイント |
|---|---|
| ① 現状分析 | 判断より先に観察 |
| ② 原因の解明 | 「意図的か」「やむを得ないか」を切り分ける |
| ③ 提案 | 敬意を払い、メリットを伝える |
| ④ 実装 | 古典学派スタイルで始める |
| ⑤ CI自動化 | テストは自動実行されてこそ輝く |

| ⑦ 見守る | 習慣が浸透するまで伴走する |

---

# 一番大事なこと

<br>

技術的なステップより先に：

<br>

既存コードベースとチームへの敬意

<br>

これがなければ、どれだけ正しい提案をしても
チームは動かない。

# Appendix

---

# 古典学派 vs ロンドン学派

| | 古典学派 | ロンドン学派 |
|---|---|---|
| モックの使用 | 極力使わない | 積極的に使う |
| テスト粒度 | 振る舞い単位 | 関数単位 |
| リファクタ耐性 | 高い | 低い（壊れやすい） |
| 偽陽性リスク | 低い | 高い |
| 導入コスト | 環境準備が必要 | 低い |

どちらが優れているかではなく、状況に応じて使い分ける。

---

# テストコードのないプロジェクトへの参画

よくある心理：

「既存コードがひどすぎる…全部書き直したい」

この気持ちはとてもわかる。
しかし、まずは既存コードへの理解と敬意が先。

<br>

コードが今の状態になった背景には
必ずそれなりの事情がある。

---

# テストを書く文化の定着に必要な時間

経験上：

- 最初の成功体験まで：1〜3ヶ月
- チームに習慣が定着するまで：6ヶ月〜1年

<br>

焦らず、着実に。
小さな成功体験の積み重ねが文化を作る。
